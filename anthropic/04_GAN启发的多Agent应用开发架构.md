# GAN 启发的多 Agent 应用开发架构

## 文章信息
- **标题**: Harness Design for Long-Running Application Development
- **发布日期**: 2026-03-24
- **链接**: https://www.anthropic.com/engineering/harness-design-long-running-apps

## 核心内容总结

这篇文章描述了一种受 GAN (生成对抗网络) 启发的多 Agent 架构,通过**将代码生成与评估分离**来改进 AI 驱动的软件开发,是解决"自我评估偏差"问题的创新方案。

### 核心架构

**三个专业化 Agent**:
1. **规划器 (Planner)**: 将简短提示扩展为详细规格
2. **生成器 (Generator)**: 增量构建应用
3. **评估器 (Evaluator)**: 测试功能并提供反馈

**设计灵感**:
- 借鉴 GAN 的生成器-判别器对抗训练思想
- 生成器专注于创造,评估器专注于批判
- 通过角色分离解决模型"自我评估偏差"问题

### 关键问题: 自我评估偏差

**问题描述**:
- 模型倾向于无批判地赞扬自己的作品
- 单 Agent 难以客观评估自己生成的代码
- 导致质量问题被忽视,迭代改进不足

**解决方案**:
- 独立的评估器 Agent 提供客观反馈
- 评估器不参与代码生成,保持批判性
- 形成生成-评估-改进的闭环

### 性能对比

**完整 Harness 版本**:
- 运行时间: 6 小时
- 成本: $200
- 产出: 功能丰富的可用应用

**单 Agent 版本**:
- 运行时间: 20 分钟
- 成本: $9
- 产出: 核心游戏机制存在缺陷

**关键洞察**:
- 22 倍的成本和 18 倍的时间换来了质量的显著提升
- 验证了"在必要时增加复杂度"的价值
- 展示了评估器在质量保证中的关键作用

## 与其他文章的关联分析

### 与基础理论的关系

**与《构建有效 Agent》(#1) 的关系**:
- **模式扩展**: 是 evaluator-optimizer 模式的深度实践和扩展
- **复杂度权衡**: $200/6小时 vs $9/20分钟 的对比完美诠释了 #1 的"仅在必要时增加复杂度"原则
- **质量导向**: 当质量比速度更重要时,复杂架构是值得的
- **三角色模式**: 规划-生成-评估是对 #1 基础模式的创新组合

### 与其他多 Agent 实践的关系

**与《多 Agent 研究系统》(#2) 的关系**:
- **架构差异**:
  - 研究系统: 主 Agent + 多个并行子 Agent (水平扩展)
  - Harness: 三个串行专业化 Agent (垂直分工)
- **协调方式**:
  - 研究系统: 主 Agent 动态协调
  - Harness: 固定的规划→生成→评估流程
- **评估方法**:
  - 研究系统: LLM-as-Judge 评估研究质量
  - Harness: 评估器 Agent 执行测试并提供反馈
- **时间尺度**: 都是长期运行任务(研究系统分钟级,Harness 小时级)

**与《C 编译器构建》(#3) 的关系**:
- **任务性质**:
  - 编译器: 结构化工程,模块化并行
  - Harness: 应用开发,迭代式串行
- **质量保证**:
  - 编译器: 单元测试和集成测试
  - Harness: 专门的评估器 Agent
- **开发模式**:
  - 编译器: 分而治之,并行开发
  - Harness: 增量构建,迭代改进

### 与评估文章的关系

**与《Agent 评估揭秘》(#6) 的关系**:
- **核心问题**: 都关注 Agent 评估的挑战
- **自我评估偏差**: Harness 通过架构设计解决了 #6 讨论的自我评估问题
- **独立评估**: 评估器 Agent 是独立评估的具体实现
- **方法论**: Harness 提供了一种实用的评估架构模式

**与《SWE-bench 评估》(#5) 的关系**:
- **自主性**: 都强调 Agent 的高度自主性
- **工具使用**: 评估器需要执行测试,类似 SWE-bench 的 Bash 执行器
- **迭代改进**: Harness 的生成-评估-改进循环类似 SWE-bench 的调试过程

**与《基础设施噪声》(#8) 的关系**:
- **长期运行**: 6 小时运行时间对基础设施稳定性要求高
- **资源需求**: 三个 Agent 同时运行需要充足的资源
- **可重现性**: 评估结果的可重现性依赖稳定的基础设施

### 与工具设计文章的关系

**与《为 Agent 编写工具》(#13) 的关系**:
- **角色专用工具**:
  - 规划器: 需要需求分析和规格编写工具
  - 生成器: 需要代码生成和文件操作工具

- **工具隔离**: 不同角色使用不同工具集,避免混淆
- **上下文效率**: 每个 Agent 只加载其需要的工具

**与《高级工具使用》(#14) 的关系**:

- **工具规模**: 应用开发可能需要大型工具库

**与《MCP 代码执行》(#15) 的关系**:
- **测试执行**: 评估器需要高效的代码执行能力
- **批量测试**: 可以批量运行多个测试用例
- **结果聚合**: 需要聚合多个测试的结果

### 与上下文管理文章的关系

**与《有效上下文工程》(#16) 的关系**:
- **角色上下文**: 每个 Agent 维护自己的上下文
- **上下文传递**: 规划器→生成器→评估器的上下文传递策略
- **上下文腐化**: 6 小时运行需要管理上下文腐化问题

**与《Claude Code 沙箱》(#19) 的关系**:
- **测试隔离**: 评估器执行测试需要沙箱隔离

- **权限控制**: 不同 Agent 可能需要不同权限

**与《Auto Mode》(#20) 的关系**:
- **自动化流程**: Harness 的固定流程适合 auto mode
- **权限审查**: 规划和生成阶段可能需要审查,评估阶段相对安全
- **效率提升**: Auto mode 可以减少 6 小时运行中的权限提示

### 与 GAN 和机器学习的关系

**设计灵感**:
- **GAN 架构**: 生成器-判别器的对抗训练
- **角色分离**: 生成和评估的职责分离
- **迭代改进**: 通过对抗训练不断提升质量

**差异**:
- GAN 是模型训练方法,Harness 是 Agent 协作架构
- GAN 的判别器是二分类器,Harness 的评估器提供详细反馈
- GAN 通过梯度更新,Harness 通过自然语言反馈

## 核心价值与影响

1. **创新架构**: 首次将 GAN 思想应用于 Agent 协作架构
2. **问题解决**: 有效解决了自我评估偏差这一关键问题
3. **质量保证**: 证明了独立评估对提升输出质量的重要性
4. **成本透明**: 清晰展示了质量提升的成本($200 vs $9)
5. **模式推广**: 规划-生成-评估三角色模式可推广到其他领域
6. **长期任务**: 展示了如何设计长期运行的 Agent 系统

## 实践启示

1. **角色分离**: 将生成和评估分离给不同 Agent 可以提升质量
2. **成本预算**: 高质量输出需要更多时间和成本投入
3. **迭代设计**: 生成-评估-改进的闭环是质量保证的关键
4. **专业化**: 每个 Agent 专注于单一职责,提高效率
5. **反馈机制**: 评估器的详细反馈是改进的基础
6. **适用场景**: 适合质量要求高、可以接受较长开发时间的项目

## 架构模式总结

| Agent 角色 | 职责 | 输入 | 输出 |
|-----------|------|------|------|
| 规划器 | 需求分析 | 用户简短提示 | 详细规格文档 |
| 生成器 | 代码实现 | 规格文档 + 评估反馈 | 应用代码 |
| 评估器 | 质量检查 | 应用代码 | 测试结果 + 改进建议 |

这篇文章为构建高质量 AI 驱动的软件开发系统提供了重要的架构参考。
